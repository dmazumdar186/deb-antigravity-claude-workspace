# App Design — 5-Principle MVP Framework

Runs **before** Phase 1. Turns a vague app idea into a written spec that Phases 1-5 can implement without re-deriving scope. Encodes Nick Saraev's app design framework (transcript: `Claude Code Mobile App Dev 1.pdf`, chapters 10, 11, 15, 25).

## Goal

Produce one Markdown spec at `C:\Users\deban\dev\mobile-apps\<slug>\APP_SPEC.md` that names:

1. The **core function** (the one thing the app would still be if you stripped everything else away).
2. The **core loop** (≤30s action→reward cycle, with at least one of: haptic, sound, visual reward).
3. **Accessory features** (3-5 max — what wraps the loop without being it).
4. A **surface-area check** (5-7 screens max — list them).
5. A **retention hook** (what brings the user back tomorrow).

This spec is what Phase 1's "App spec" input expects. A spec produced here also feeds the per-app `CLAUDE.md` generated by `/init` after Phase 1 ships locally.

## Inputs

- App slug (kebab-case, from `bootstrap_mobile_app.py`)
- One paragraph of user intent: what the app does and who it's for
- Optional: screenshots of competitor apps for design inspiration (chapter 28 — pull padding/corner-radius/color cues, not flows)

## Tools/Scripts

- None deterministic for spec authoring — this is a Claude reasoning pass with the user. Output is a Markdown file.
- Voice transcription (Aqua or similar) is the recommended input method per Nick — long-form spoken intent gets richer specs than typed prompts.
- **`execution/mobile_apps/app_store_research.py`** — optional competitive ASO research before drafting the spec. Scrapes App Store and/or Play Store search results for a competitor keyword. Runs in parallel for multi-competitor batches.

  Single competitor:
  ```
  py execution/mobile_apps/app_store_research.py --query "Headspace" --store appstore
  ```

  Multiple competitors (fan-out, parallel scraping, 8 workers):
  ```
  py execution/mobile_apps/app_store_research.py \
    --competitors-file competitors.csv \
    --stores ios android \
    --max-workers 8
  ```

  `--max-workers N` (default 8) controls ThreadPoolExecutor concurrency. Keep ≤ 8 to stay within Firecrawl rate limits. For 5+ competitors across both stores, use the Dynamic Workflow at `.claude/workflows/aso-research.md` instead — it fans out to 16 parallel Haiku sub-agents and cuts wall-clock from ~10 min to ~75s.

## Outputs

- `C:\Users\deban\dev\mobile-apps\<slug>\APP_SPEC.md` with the 5 sections below
- One-line summary written to `execution/mobile_apps/registry.json`'s `<slug>.spec_summary` field
- Git commit on the app repo's `main` branch: `chore: app spec (5-principle framework)`

## Steps

1. **Read the user's intent paragraph.** If the paragraph is < 3 sentences or under-specifies the domain, ask exactly one batched clarifying question (use AskUserQuestion with 2-4 options).
2. **Draft section 1 — Core function.** Pose the test: "If I removed every feature except one, which one *must* remain for this to be the app?" Write it as a single sentence. If the user wrote down 3 things, pick the load-bearing one; the other two move to accessory features.
3. **Draft section 2 — Core loop.** Specify (a) the user action, (b) the reward, (c) the latency between them (must be ≤30s for retention to work). Pick at least one of: haptic feedback, sound chime, visual animation (Reanimated for confetti/burst is Nick's default; flag CPU cost if vision-heavy). For games tighten to ≤5s; for utility apps ≤30s is fine.
4. **Draft section 3 — Accessory features.** List 3-5 features that *support* the loop without changing the core function. Common patterns from Nick: history/log view, chart/graph, customization, multi-tier variant (e.g., "1x/day" vs "Nx/day" habits), social. Mark each as P0 (Phase 1) or P1 (Phase 2+).
5. **Draft section 4 — Surface-area check.** Enumerate every screen. **Hard cap: 7.** If the user wants more, force them to merge or move to P2. Typical layout: onboarding → today/main → history → manage → settings → 1-2 accessory screens.
6. **Draft section 5 — Retention hook.** Pick one: N-day challenge (Nick's default — 3 days minimum), daily check-in nudge, scheduled push notification, streak/gem-style escalation (cf. Opal). Note that push notifications require Phase 4 backend (Supabase or CF Worker) — design for it in Phase 1 even if implementation defers.
7. **Surface-area sanity pass.** Re-read sections 3 + 4. If accessory features imply screens you didn't enumerate, either drop the feature or add the screen and re-check the 7-cap.
8. **Write `APP_SPEC.md`** to `C:\Users\deban\dev\mobile-apps\<slug>\APP_SPEC.md`. Template:
   ```markdown
   # <slug> — App Spec
   ## Core Function
   <one sentence>
   ## Core Loop
   - Action: <...>
   - Reward: <...>
   - Latency: <≤30s>
   - Sensory: [haptic | sound | visual | combo]
   ## Accessory Features
   - P0: <feature> — <one-line how it supports the loop>
   - P1: <feature>
   ## Surface Area
   1. <Screen 1>
   2. <Screen 2>
   ...
   <≤7 total>
   ## Retention Hook
   <type>: <description>; backend dependency: <none | Phase 4>
   ## Inspiration (optional)
   <competitor app(s) for design cues only — not flows>
   ```
9. **Update registry.** Append a `spec_summary` field to the slug's entry in `execution/mobile_apps/registry.json` with the one-line core function.
10. **Commit.** `git -C C:\Users\deban\dev\mobile-apps\<slug> add APP_SPEC.md && git -C ... commit -m "chore: app spec (5-principle framework)"`.
11. **Hand-off.** Print the spec path to the user and recommend `/mobile-app phase 1 --app <slug>`. The Phase 1 agent reads `APP_SPEC.md` directly.

## After Phase 1 ships — run `/init`

Per Nick chapter 11: once the local MVP boots in Expo Go, the app has enough structure that a hand-written or template-copied `CLAUDE.md` is suboptimal. Have the user run `/init` inside the app repo to generate a per-app `CLAUDE.md` from the actual code. This replaces the template's placeholder file with one Claude wrote knowing the real state of `src/`, `package.json`, and the navigation tree.

```
cd C:\Users\deban\dev\mobile-apps\<slug>
/init
```

Then `/clear` to reset token usage before Phase 2.

## Edge Cases

- **User wants > 7 screens.** Push back once with: "Nick's surface-area check caps at 7. Which can move to P1 or merge?" If they insist, write it but flag in the spec under `## Notes` that the cap was overridden — this becomes a risk to reference during testing.
- **Core loop latency > 30s.** Usually means the reward is misplaced. Probe: is there a sub-step that *could* be rewarded immediately? Most "slow" loops have a faster sub-loop hidden inside.
- **No retention hook the user is happy with.** Common for utility apps. Acceptable to ship without one for v1 but mark as a risk; retention apps without hooks churn at >90% by week 2 (industry baseline).
- **Vision/AI app (Cal AI clone, receipt scanner).** Core loop usually = (take photo → AI returns result → log it). Reward is the result itself; surface area shrinks to 3-4 screens. Phase 5b vision pipeline plugs in here.
- **Spec drift mid-Phase-1.** If the implementing agent finds the spec impractical (e.g., the chosen state shape can't represent the surface area), update `APP_SPEC.md` *before* fixing the code, not after. The spec is the source of truth.
- **User wants to skip this directive and go straight to Phase 1.** Allowed, but Phase 1's "App spec" input then comes from the user verbally — quality is lower and rework is more likely. Flag this once and proceed.
