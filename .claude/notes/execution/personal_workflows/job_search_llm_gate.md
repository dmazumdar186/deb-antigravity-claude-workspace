# job_search_llm_gate.py Notes

- [technical] Gemini 2.5 thinking-token truncation: Gemini 2.5 models (flash / flash-lite) use a thinking budget that can consume the majority of `max_output_tokens`, leaving little room for the actual JSON response. This causes the model to truncate mid-JSON, producing unparseable output. Workaround: set `response_mime_type: application/json` in the generation config AND cap `max_output_tokens` at 8192. The JSON MIME type forces the model to prioritize the structured output over thinking tokens, and the 8192 cap prevents runaway thinking budget consumption.
- [technical] response_mime_type workaround: When calling Gemini with structured JSON output (e.g. `GateVerdict` schema), always pass `generation_config={"response_mime_type": "application/json", "max_output_tokens": 8192}`. Without `response_mime_type`, Gemini 2.5 may wrap JSON in markdown fences or truncate at the thinking token boundary, causing json.loads() to raise. This applies to both `gemini-2.5-flash` and `gemini-2.5-flash-lite`.
- [technical] throttle_s=7 rationale: The Gemini free tier allows ~10 RPM (requests per minute) for Gemini 2.5 Flash. With batch_size=10 jobs per classify_batch() call, a 7-second sleep between batches keeps the effective RPM at ~8–9 (10 batches/70s ≈ 8.6 RPM), staying safely under the limit with headroom for retries. Reducing throttle_s below 6 risks 429 rate-limit errors on back-to-back job sheet runs.
